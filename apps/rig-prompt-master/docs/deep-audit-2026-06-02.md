## What Was Supposed To Be Built

The production plan called for one canonical product named RIG Master Prompter:

- A usable local/web workbench for prompt intake, target mode, context selection, audience done model, approvals, agent runs, and ProofPacket recall.
- A macOS application that opens the same product, not a separate fake app.
- A Next.js/Vercel-ready API surface for prompt runs, context sync, agent runs, approvals, ProofPackets, v15 catalog, v10 readiness, v25 audit, audience model, and hardening status.
- Hybrid local/cloud context connectors for GitHub, Gitea, QNAP, Recall.it, uploads, local repos, and approved web sources.
- Approval gates before repo writes, browser submits, account changes, external sends, private exports, or destructive actions.
- ProofPacket evidence for meaningful operations.
- User, QA, navigation, API, build, and desktop launch testing with inspectable proof.

## What Was Actually Built

### Built And Locally Operational

- Next.js app shell under `apps/rig-prompt-master`.
- Local API routes under `apps/rig-prompt-master/app/api/v1`.
- Local JSON-store-backed prompt runs, context source registry, approvals, worker jobs, agent-run records, and ProofPacket records.
- v15 catalog route, v10 readiness route, v25 audit route, audience done model route, and hardening route.
- Prompt run creation that returns a repaired prompt, DoneContract, citations, gates, selected v15 questions, score, and ProofPacket id.
- Native macOS WebView wrapper source under `desktop/`, plus desktop build tooling.
- Local verifier and unit tests.
- Playwright navigation audit added in this pass.

### Built But Partial

- Context connector boundaries exist for GitHub, Gitea, QNAP, Recall.it, uploads, local repos, and web sources, but deep credential-backed sync is not fully proven by this audit.
- Approval-gated agent-run workflow exists as a local bounded state machine, but real long-running browser/repo agents are not production-proven here.
- API-key/dev auth paths exist, but SSO/OIDC and production workspace roles are not proven in this local audit.
- ProofPacket records exist and can be recalled, but signing, hash-chain integrity, and external verifier manifests are not complete.
- Desktop wrapper is real, but Apple notarization/signing proof is not complete.

### Not Built Or Not Proven

- Live end-to-end GitHub, Gitea, QNAP, Recall.it, and web scraping integration tests with real configured accounts.
- pgvector runtime retrieval with production embeddings and citations quality evaluation.
- Vercel production deployment smoke test with managed Postgres.
- SSO-first production auth with workspace roles.
- Docker worker service for long-running browser/repo agents.
- Independent verifier service.
- Accessibility audit with axe/Lighthouse proof.
- Visual regression suite beyond the current Playwright screenshots.
- Apple notarized release build.

## What Was Fake Or Scaffolded Before This Fix

These are not theoretical issues; they were found by inspecting the source and then confirmed by the initial behavior Mike reported.

- Sidebar nav buttons had no click handlers.
- Workbench was hardcoded as the active nav item.
- `Context`, `Personas`, `Runs`, `Proof`, `Agents`, `Approvals`, `Gate Checklist`, `Audit Log`, `Catalog v15`, `Connectors`, `Integrations`, and `Settings` did not show real section views.
- `Collapse` did not collapse the sidebar.
- `+ Add Source`, `View Run Details`, `All Personas`, Proof tabs, and API Recall copy were visible but inert.
- API Recall showed a hardcoded URL that could point at the wrong port instead of the running app origin.
- There was no committed Playwright test proving the navigation worked.

## What Was Fixed On 2026-06-02

- Added real sidebar application state and section rendering for all 13 nav items.
- Added dynamic headings and section subtitles for each route-like view.
- Added working section views for context, personas, runs, proof, agents, approvals, gate checklist, audit log, catalog, connectors, integrations, and settings.
- Made `Collapse` toggle the sidebar.

- Made `View Run Details` route to Runs.

## Mac Window Behavior

The desktop wrapper is a normal macOS app window, so it can be moved around the screen. That is expected for a standard Mac app and is usually desirable if the app is meant to live in the Dock.

If the requirement is a fixed-position kiosk, menu-bar, floating palette, or non-movable window, that is a separate product requirement and has not been implemented. It should be decided explicitly because it changes normal Mac user expectations.
